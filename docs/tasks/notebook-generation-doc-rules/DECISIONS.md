# notebook-generation 文档重写 决策说明

## 1. 任务目标
本任务的目标不是修改 Wolfram 算法，也不是修改 notebook 导出脚本，而是重写仓库级规则文档 `docs/notebook-generation.md`，使其明确满足以下条件：

1. `docs/` 目录只保存文档，不保存实现代码。
2. notebook 的生成链路仍然与当前仓库实现一致。
3. 文档职责、源码职责、脚本职责与导出产物职责必须边界清晰。

## 2. 决策背景
当前仓库同时包含以下几类对象：

1. `docs/` 中的规则、决策与任务文档。
2. `wl/` 中的 Wolfram 源文件。
3. `scripts/` 中的 PowerShell 入口脚本。
4. `notebooks/` 与 `notebooks/exports/` 中的 notebook 文件与导出结果。

如果规则文档本身继续强调“在文档中承载结构化源码示例”或让读者误以为说明源码应放入 `docs/`，就会带来两个问题：

1. 目录职责混淆，后续维护时容易把文档目录误用为实现目录。
2. 稳定规则无法清楚回答“应修改哪里”和“哪些内容只是说明，哪些内容才是实际执行入口”。

因此，本次决策是把 `docs/notebook-generation.md` 改写为纯规则文档，重点阐明目录职责与生成链路，而不是继续扩展源码示例。

## 3. 与当前实现保持一致的部分
本次重写保留了当前仓库已经稳定的实现边界：

1. 运行 `.wl` 或 `.wls` 仍然优先使用 `scripts/run_wl.ps1`。
2. 生成或检查 `.nb` 仍然优先使用 `scripts/export_nb.ps1`。
3. `ScriptOutput`、`PackageEditorInput`、`StructuredSource` 三种模式的适用条件保持不变。
4. `PackageEditorInput` 与 `StructuredSource` 仍依赖 Wolfram Front End。

也就是说，这次不是改实现，而是把文档修正为正确描述现有实现。

## 4. 为什么不再保留大段源码式示例
从工程治理角度看，规则文档的职责是说明：

1. 目录应该如何分工。
2. 何时使用哪条生成链路。
3. 修改 notebook 时应回到哪个真实源文件。
4. 哪些做法是不推荐的。

而源码示例的职责属于实现层，应放在 `wl/`、`scripts/`、`notebooks/` 或任务目录中。若把大量源码式内容放在 `docs/` 中，读者容易把说明误当成实际入口，反而降低可维护性与可复现性。

## 5. 科学背景、数学知识与物理理论说明
本任务属于仓库工程规则文档重写，不直接研究某个数学模型或物理问题，因此：

1. 不涉及新的数学公式推导。
2. 不涉及新的物理理论假设。
3. 不涉及对 Wolfram 数值计算模型的科学结论修正。

但它仍然服务于科学计算工作流。其工程意义在于保证以下研究过程更可复现：

1. 源码放在源码目录。
2. 运行入口放在脚本目录。
3. 结果放在导出与结果目录。
4. 规则与说明放在文档目录。

这种“单一真实来源”与“目录职责分离”的做法，有助于减少 notebook 研究工作中常见的版本漂移、说明与实现脱节、导出链路不可追踪等问题。

## 6. 当前现状与遗留问题
当前现状：

1. 仓库已有稳定的 notebook 生成入口。
2. `docs/notebook-generation.md` 已被改写为纯文档规则。
3. 文档已明确 `docs/` 只保存文档，不保存代码。

遗留问题：

1. `notebooks/AGENTS.md` 中仍保留一份近似副本，后续若继续演化规则，需要决定是否同步收敛其内容。
2. 其他历史任务文档中若存在“文档目录职责”表达不清的情况，后续也可做一致性清理。
3. 当前终端环境对部分中文文件显示存在编码噪声，后续可单独检查文档编码与查看工具链的一致性。

## 7. 后续发展方向
1. 将目录职责类规则进一步拆分为“文档规范”“notebook 规范”“任务归档规范”三类专门文档，减少单文档负担。
2. 在后续任务中逐步清理历史文档里可能引起误解的旧表述。
3. 如有必要，可增加一份“目录职责总览”文档，集中说明 `docs/`、`wl/`、`scripts/`、`notebooks/`、`results/` 的边界。
