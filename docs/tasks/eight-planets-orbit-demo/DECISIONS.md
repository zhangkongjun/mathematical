# 八大行星运行轨迹研究课题 决策说明

## 1. 文档定位
本文档用于说明 `eight-planets-orbit-demo` 的科学背景、问题定义、数学近似、工程实现决策、当前限制与后续扩展方向。它既服务于本仓库内的持续维护，也服务于后续研究复盘、教学说明，以及与外部研究助手继续沟通时的背景交接。

## 2. 科学背景与问题定义
太阳系八大行星绕日运动，是经典天体力学、科学可视化和计算教学中最常见的入门课题之一。对该课题的实现通常分为两类：

1. 以教学、展示和概念解释为目标的近似模型。
2. 以观测、拟合和数值精度为目标的高精度星历模型。

本任务明确选择第一类路线。当前课题的核心问题不是“如何得到观测级精度的行星位置”，而是“如何在 Wolfram / Mathematica 工作流中，构建一个可复现、可讲解、可导出 notebook、并且便于后续扩展的八大行星运行轨迹演示模型”。

因此，本课题的直接目标定义为：

1. 用单一最终 `.wl` 源文件表达八大行星运行轨迹演示的说明与实现。
2. 通过仓库级默认脚本稳定生成可阅读、可调试、可演示的 `.nb`。
3. 在不引入复杂星历依赖的前提下，展示八大行星的轨道层级、相对公转快慢和多尺度观察效果。

本课题的非目标同样需要明确：

1. 当前版本不是严格的太阳系动力学求解器。
2. 当前版本不用于观测级预测、轨道拟合、误差传播分析或长期稳定性研究。
3. 当前版本不试图替代 JPL DE、`AstronomicalData` 或其他高精度星历方案。

## 3. 当前稳定方案的总决策
围绕上面的目标，当前实现收敛为以下稳定决策：

1. 使用单个结构化 `.wl` 文件同时承载说明单元和最终可执行代码，避免“文档一份、代码一份、notebook 再拼一份”导致的维护漂移。
2. 最终 notebook 统一通过仓库级 `scripts/export_nb.ps1 -GenerationMode StructuredSource` 生成，不再走“先生成 `.nb` 再手工解析和回写 `.nb`”的旧链路。
3. 动力学层采用二维、共面、日心、圆轨道、匀角速度的教学近似。
4. 视觉表达层允许使用非物理比例增强，例如标签偏移、行星显示半径增强、尾迹长度控制和多档视野裁剪。
5. 所有稳定规则都要求回写到任务文档中，确保聊天记录之外仍有完整、正式、可独立阅读的说明材料。

## 4. 数学模型、物理理论与核心概念
### 4.1 采用的轨道近似
当前模型把每颗行星视为位于二维平面中的圆轨道上，并假设太阳固定在原点。设：

1. `r` 表示该行星轨道半径，单位近似取天文单位 AU。
2. `T` 表示该行星公转周期，单位取地球年。
3. `t` 表示动画中的模拟时间，单位同样取年。
4. `theta(t)` 表示行星在时刻 `t` 的轨道角位置。
5. `omega` 表示角速度。

则当前模型使用

`theta(t) = 2πt / T`

并进一步得到

`omega = d theta / dt = 2π / T`

在该近似下，行星在平面中的位置写作

`x(t) = r cos(theta(t))`

`y(t) = r sin(theta(t))`

或者向量形式

`r_vec(t) = (r cos(theta(t)), r sin(theta(t)))`

该模型的适用条件是：

1. 目标以概念解释、轨道层级展示和交互式观察为主。
2. 可以接受忽略轨道偏心率、倾角、岁差、摄动和真实历元。
3. 更关心内外行星“谁更快、谁更远、尺度差有多大”，而不是某个日期上的精确位置。

### 4.2 与开普勒定律的关系
虽然实现层采用了圆轨道和匀角速度近似，但理解该模型仍应放在经典天体力学背景下：

1. 开普勒第一定律指出，真实行星轨道更接近椭圆，而不是完美圆。
2. 开普勒第二定律指出，行星在轨道上并非匀速转动，而是满足面积速度守恒，即 `dA/dt = 常数`。
3. 开普勒第三定律建立了轨道半长轴与公转周期之间的关系；对于绕同一中心天体运动的轨道，通常可写为 `T^2 ∝ a^3`。

当前项目并没有直接数值求解椭圆轨道或面积速度守恒，而是把上述理论简化为“用平均轨道半径和平均公转周期驱动一个教学近似模型”。这意味着它保留了行星远近与快慢的大体层级关系，但舍弃了真实轨道的精细结构。

### 4.3 与牛顿引力理论的关系
真实的绕日运动可追溯到万有引力与二体问题。其经典形式为

`F = G m1 m2 / r^2`

其中：

1. `G` 为万有引力常数。
2. `m1` 与 `m2` 为两个天体质量。
3. `r` 为两者间距离。

若进一步考虑多体摄动、轨道元素随时间变化、坐标系转换和历元对齐，则必须进入更复杂的天体力学和星历计算框架。当前项目不走该路线，而只保留其教学上最直观的一层结果，即“不同轨道半径和周期下的绕日相对运动”。

### 4.4 轨迹尾迹与视野裁剪
为了让动画更容易被观察，当前实现还引入两类数学上简单但视觉上有效的处理：

1. 尾迹采样：在区间 `[t - Δt, t]` 上选取离散采样点 `tau_i`，连接 `r_vec(tau_i)`，得到一小段最近轨迹。
2. 视野裁剪：对给定视野半径 `z`，只显示满足 `r <= z` 的行星及其轨道。

尾迹的作用是增强“运动正在发生”的可读性；视野裁剪的作用是缓解太阳系尺度差过大带来的拥挤问题。

### 4.5 视觉编码中的非线性缩放
真实行星半径差异和轨道尺度差异都很大。若直接按照真实比例显示，外行星在全景视图中仍可能难以辨认。因此当前实现采用温和的对数增强：

`R_draw = 0.045 + 0.012 ln(1 + r)`

其中：

1. `R_draw` 为图上绘制的行星圆盘半径。
2. `r` 为轨道半径近似值。

这不是物理半径模型，而是视觉编码策略。其目的是在不完全丢失“内外有别”的前提下，提高 notebook 演示时的可辨识度。

## 5. 工程实现与代码组织决策
### 5.1 单一事实来源
本任务最重要的工程决策，是采用“说明与代码同源”的结构化 `.wl` 文件。这样做的原因包括：

1. notebook 文案、代码和最终导出结果可以追溯到同一个源文件。
2. 避免直接重写 `.nb` 文本导致的中文说明损坏问题。
3. 更符合本仓库“不要直接修改最终 `.nb`，优先修改源文件再重新导出”的稳定规则。

### 5.2 参数集中管理
`planetData` 集中维护每颗行星的名称、颜色、轨道颜色、轨道半径、公转周期与标签偏移。这样做的好处是：

1. 行星参数层与绘图逻辑层分离。
2. 调整教学视觉效果时，不必在多个函数中重复修改。
3. 后续若引入外部数据源，可以优先替换这层参数组织，而不必立即重写渲染逻辑。

### 5.3 交互表达优先于精细输出
最终表达选择 `Manipulate` 作为交互入口，而不是一开始就导出固定帧序列或视频。这一决策基于以下考虑：

1. 交互式拖动更适合教学与讨论。
2. notebook 用户可以直接观察不同 `zoom` 档位下的轨道层级。
3. 在模型仍处于近似演示阶段时，交互式验证比批量渲染更有价值。

## 6. 当前研究现状、已知限制与遗留问题
### 6.1 当前研究现状
从更大的课题背景看，太阳系轨道可视化通常处于以下几类实现谱系中：

1. 纯教学近似模型：强调概念直观、交互顺滑和公式可解释性。
2. 半真实数据驱动模型：使用公开轨道要素或星历参数，兼顾可视化和一定精度。
3. 高精度星历模型：依赖成熟数据源、坐标系统和时间系统，关注误差控制与可验证性。

本任务目前稳定停留在第一类和第二类之间，更偏向第一类。它已经具备规范的源文件、导出链路和文档体系，但在物理真实性上仍保持克制，不主动跨越到高精度星历计算。

### 6.2 已知限制
当前实现至少存在以下限制：

1. 轨道形状被简化为圆，而非椭圆。
2. 所有轨道被简化为共面二维结构，没有轨道倾角和节点信息。
3. 角速度按平均周期恒定给出，不体现近日点和远日点附近的速度变化。
4. 初始相位没有绑定真实历元，因此动画起点不是某个具体日期下的真实太阳系快照。
5. 行星可视大小、标签偏移和尾迹长度都带有人为视觉优化，不可解释为真实物理尺度。
6. 当前视角是理想化日心平面视图，而不是观测者视角或天球投影视角。
7. 未纳入多体摄动、共振、岁差、黄道倾角、卫星系统或小天体影响。
8. `Manipulate` 的流畅度会受到 Front End 状态、机器性能和图层复杂度影响。

### 6.3 当前遗留问题
即使在教学近似框架内，仍有一些尚未完成但值得保留的问题：

1. 视觉标签在不同中文字体环境下可能出现重叠，尚缺少自动布局策略。
2. 参数当前内嵌在源文件中，尚未抽成独立数据文件或可替换数据层。
3. 尚未建立“教学增强版”和“更真实物理版”的双模式对照输出。
4. 尚未形成统一的批量导出图像、GIF 或短视频入口。

## 7. 未来发展方向、可扩展实现路径与待验证假设
### 7.1 扩展路径
后续如果要从教学演示进一步走向研究或更真实的模拟，可按以下路径扩展：

1. 把圆轨道替换为椭圆轨道，并显式引入偏心率、半长轴、近心点角等轨道要素。
2. 增加轨道倾角、升交点经度和三维投影，得到更真实的空间结构。
3. 将初始相位绑定到明确历元，使动画与某个具体日期对齐。
4. 引入 `AstronomicalData`、JPL DE 或其他外部数据源，形成数据驱动的轨道状态初始化。
5. 将参数层、动力学层、渲染层和 notebook 导出层进一步解耦，方便单独测试和替换。
6. 增加速度矢量、近日点/远日点标记、会合事件标记或多参考系对比。

### 7.2 值得继续验证的假设
本任务后续最值得验证的假设包括：

1. 在教学可读性不显著下降的情况下，是否可以引入轻量级椭圆轨道参数而保持交互流畅。
2. 是否可以通过简单的标签避让策略显著改善中文环境下的多尺度标注质量。
3. 是否可以用同一份结构化源文件同时生成“课堂演示版”和“研究说明版”两种 notebook。
4. 若采用外部轨道数据源，是否仍能保留当前“说明与代码同源”的维护优势。

## 8. 维护建议
为了让本课题在后续迭代中保持清晰，建议遵守以下原则：

1. 只要最终 notebook 的结构、文案、公式或说明发生变化，就应优先修改 `.wl` 源文件后重新导出。
2. 若只调整显示效果，也应同步更新本任务文档中关于适用范围和限制的描述。
3. 若将来引入更真实的轨道模型，必须明确区分“教学近似”与“高精度计算”，避免二者混淆。
4. `DECISIONS.md`、`ITERATIONS.md` 与 `FINAL_FILES.md` 应始终与当前最终 `.wl`、任务脚本和 notebook 导出链路保持一致。
